## 第一部分：核心理念

### 1.1 什么是LLM-Wiki？

#### 定义

**LLM Wiki** 是一种使用大语言模型（LLM）构建和维护**个人知识库**的创新模式，由Andrej Karpathy于2026年4月提出。

#### 核心思想

> 让AI持续编译和维护你的知识库，而不是每次查询时重新检索

---

### 1.2 传统RAG的局限性

#### 传统RAG工作流程

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ 上传文档 │────▶│ 向量索引 │────▶│ 检索片段 │────▶│ 拼凑答案 │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
```

#### 核心问题

| 问题 | 说明 |
|------|------|
| ❌ **无积累** | 每次查询从头开始 |
| ❌ **无综合** | 只能拼凑片段，无法形成整体认知 |
| ❌ **无记忆** | 好的答案消失在聊天历史中 |
| ❌ **无进化** | 知识库不会随时间改进 |
| ❌ **无连接** | 文档之间的关系需要每次重新发现 |

#### 实际体验

```
第1次提问："Transformer架构的原理？"
    → LLM检索论文片段 → 回答

第2次提问："Transformer vs RNN的优缺点？"  
    → 又一次检索 → 又一次拼凑 → 回答

第3次提问："BERT是如何使用Transformer的？"
    → 再一次从头开始...

❌ 之前的推理过程完全丢失
❌ 知识之间的连接需要重新建立
```

### 1.3 LLM-Wiki的创新之处

#### 工作流程

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ 上传文档 │────▶│ AI阅读  │────▶│ 编译Wiki │────▶│ 持久维护 │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
      ↑                                              │
      └──────────────── 更新与扩展 ◀─────────────────┘
```

#### 核心优势

| 优势 | 说明 |
|------|------|
| ✅ **知识编译** | 文档被编译成结构化Wiki页面 |
| ✅ **持续积累** | 每个新文档让Wiki更丰富 |
| ✅ **自动维护** | AI自动更新、链接、检查矛盾 |
| ✅ **复利增长** | 知识库价值随时间指数增长 |
| ✅ **交叉引用** | 自动建立和维护文档间连接 |

| 特性 | 说明 |

**第二层：Wiki层（The Wiki）**

| 特性 | 说明 |
|------|------|

**第三层：Schema/工具层（The Schema/Tools）**

| 特性 | 说明 |
|------|------|
| **定位** | 治理与工具支持 |
| **内容** | 规则文档、索引、日志、搜索引擎、CLI工具 |
| **权限** | 人类定义规则，LLM遵循 |
| **作用** | 确保一致性和可扩展性 |

### 1.5 四大核心操作

#### 操作流程图

```
┌──────────┐
│  Ingest  │  导入新资料
│  (导入)  │────────────┐
└──────────┘            │
                        ↓
┌──────────┐       ┌─────────┐
│  Query   │       │  Wiki   │
│  (查询)  │◀──────│  知识库  │
└──────────┘       └─────────┘
     │                   ↑
     │ (File Back)       │
     └───────────────────┘
                        ↑
┌──────────┐            │
│   Lint   │  健康检查   │
│  (自检)  │────────────┘
└──────────┘
```

---

#### 1. Ingest（导入）

**流程**：

```
Step 1: 上传文档到原始资料集合
    ↓
Step 2: LLM读取并与你讨论关键要点
    ↓
Step 3: 生成文档摘要页面
    ↓
Step 4: 更新索引文件
    ↓
Step 5: 创建或更新相关实体页、概念页
    ↓
Step 6: 标记与现有知识的矛盾
    ↓
Step 7: 在日志中记录本次导入
```

**影响范围**：单个文档可能触及10-15个Wiki页面

---

#### 2. Query（查询）

**流程**：

```
Step 1: 用户提问
    ↓
Step 2: LLM搜索相关Wiki页面（通过索引或搜索工具）
    ↓
Step 3: 读取相关页面
    ↓
Step 4: 综合答案并附上引用
    ↓
Step 5: (可选) 将好答案归档回Wiki
```

**输出形式**：
- Markdown文档
- 比较表格
- 幻灯片（Marp）
- 图表（matplotlib）
- Canvas画布

---

#### 3. Lint（自检）

**检查项目**：

| 检查内容 | 说明 | 自动化 |
|---------|------|-------|
| 🔍 矛盾检查 | 页面之间的矛盾声明 | ✅ |
| 📅 过时声明 | 被新资料取代的信息 | ✅ |
| 🚫 孤儿页面 | 没有入站链接的页面 | ✅ |
| 🏷️ 缺失页面 | 被提及但未创建的概念 | ✅ |
| 🔗 缺失交叉引用 | 应该存在的链接 | ✅ |
| 🌐 数据空白 | 可通过网络搜索填补的空白 | ⚠️ |

---

#### 4. File Back（回填）

**理念**：

> 好的答案不应该消失在聊天历史中

**机制**：

```
优质查询答案 → 归档为新Wiki页面 → 成为知识库一部分 → 未来查询可引用
```

**结果**：

```
你的探索过程本身也在为知识库复利增长
```

### 2.6 优缺点分析

#### 优势

| 劣势 | 说明 |
|------|------|
| ❌ **无Web界面** | 依赖Obsidian查看 |

| ❌ **移动端受限** | Obsidian移动版功能有限 |
| ❌ **缺少云同步** | 需手动配置（如iCloud、Dropbox） |
