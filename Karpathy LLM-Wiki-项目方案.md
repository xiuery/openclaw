## 第一部分：核心理念

### 1.1 什么是LLM-Wiki？

#### 定义

**LLM Wiki** 是一种使用大语言模型（LLM）构建和维护**个人知识库**的创新模式，由Andrej Karpathy于2026年4月提出。

#### 核心思想

> 让AI持续编译和维护你的知识库，而不是每次查询时重新检索

---

### 1.2 传统RAG的局限性

#### 传统RAG工作流程

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ 上传文档 │────▶│ 向量索引 │────▶│ 检索片段 │────▶│ 拼凑答案 │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
```

#### 核心问题

| 问题 | 说明 |
|------|------|
| ❌ **无积累** | 每次查询从头开始 |
| ❌ **无综合** | 只能拼凑片段，无法形成整体认知 |
| ❌ **无记忆** | 好的答案消失在聊天历史中 |
| ❌ **无进化** | 知识库不会随时间改进 |
| ❌ **无连接** | 文档之间的关系需要每次重新发现 |

#### 实际体验

```
第1次提问："Transformer架构的原理？"
    → LLM检索论文片段 → 回答

第2次提问："Transformer vs RNN的优缺点？"  
    → 又一次检索 → 又一次拼凑 → 回答

第3次提问："BERT是如何使用Transformer的？"
    → 再一次从头开始...

❌ 之前的推理过程完全丢失
❌ 知识之间的连接需要重新建立
```

---

### 1.3 LLM-Wiki的创新之处

#### 工作流程

```
┌─────────┐     ┌─────────┐     ┌─────────┐     ┌─────────┐
│ 上传文档 │────▶│ AI阅读  │────▶│ 编译Wiki │────▶│ 持久维护 │
└─────────┘     └─────────┘     └─────────┘     └─────────┘
      ↑                                              │
      └──────────────── 更新与扩展 ◀─────────────────┘
```

#### 核心优势

| 优势 | 说明 |
|------|------|
| ✅ **知识编译** | 文档被编译成结构化Wiki页面 |
| ✅ **持续积累** | 每个新文档让Wiki更丰富 |
| ✅ **自动维护** | AI自动更新、链接、检查矛盾 |
| ✅ **复利增长** | 知识库价值随时间指数增长 |
| ✅ **交叉引用** | 自动建立和维护文档间连接 |

### 1.4 三层架构设计

#### 架构图

```
┌────────────────────────────────────────────────────────┐
│                 第一层：Raw Sources                     │
│        (PDF、文章、笔记、代码、图片、视频转录)          │
│              不可变的真相之源（Truth Layer）            │
└──────────────────────┬─────────────────────────────────┘
                       ↓
┌────────────────────────────────────────────────────────┐
│                  第二层：The Wiki                       │
│      (AI生成的Markdown页面、摘要、实体页、概念页)       │
│             由LLM完全控制（Knowledge Layer）            │
└──────────────────────┬─────────────────────────────────┘
                       ↓
┌────────────────────────────────────────────────────────┐
│               第三层：The Schema/Tools                  │
│         (规则文档、索引、日志、搜索工具)                │
│            指导LLM行为（Governance Layer）              │
└────────────────────────────────────────────────────────┘
```

---

#### 层级详解

**第一层：原始资料层（Raw Sources）**

| 特性 | 说明 |
|------|------|
| **定位** | 不可变的真相之源 |
| **内容** | PDF、网页、笔记、代码、图片等 |
| **权限** | 只读，LLM从不修改 |
| **作用** | 提供原始事实依据 |

**第二层：Wiki层（The Wiki）**

| 特性 | 说明 |
|------|------|
| **定位** | AI维护的知识网络 |
| **内容** | 摘要、实体页、概念页、比较页、综合分析 |
| **权限** | LLM完全控制，自动更新 |
| **作用** | 提供结构化、互联的知识表示 |

**第三层：Schema/工具层（The Schema/Tools）**

| 特性 | 说明 |
|------|------|
| **定位** | 治理与工具支持 |
| **内容** | 规则文档、索引、日志、搜索引擎、CLI工具 |
| **权限** | 人类定义规则，LLM遵循 |
| **作用** | 确保一致性和可扩展性 |

### 1.5 四大核心操作

#### 操作流程图

```
┌──────────┐
│  Ingest  │  导入新资料
│  (导入)  │────────────┐
└──────────┘            │
                        ↓
┌──────────┐       ┌─────────┐
│  Query   │       │  Wiki   │
│  (查询)  │◀──────│  知识库  │
└──────────┘       └─────────┘
     │                   ↑
     │ (File Back)       │
     └───────────────────┘
                        ↑
┌──────────┐            │
│   Lint   │  健康检查   │
│  (自检)  │────────────┘
└──────────┘
```

---

#### 1. Ingest（导入）

**流程**：

```
Step 1: 上传文档到原始资料集合
    ↓
Step 2: LLM读取并与你讨论关键要点
    ↓
Step 3: 生成文档摘要页面
    ↓
Step 4: 更新索引文件
    ↓
Step 5: 创建或更新相关实体页、概念页
    ↓
Step 6: 标记与现有知识的矛盾
    ↓
Step 7: 在日志中记录本次导入
```

**影响范围**：单个文档可能触及10-15个Wiki页面

---

#### 2. Query（查询）

**流程**：

```
Step 1: 用户提问
    ↓
Step 2: LLM搜索相关Wiki页面（通过索引或搜索工具）
    ↓
Step 3: 读取相关页面
    ↓
Step 4: 综合答案并附上引用
    ↓
Step 5: (可选) 将好答案归档回Wiki
```

**输出形式**：
- Markdown文档
- 比较表格
- 幻灯片（Marp）
- 图表（matplotlib）
- Canvas画布

---

#### 3. Lint（自检）

**检查项目**：

| 检查内容 | 说明 | 自动化 |
|---------|------|-------|
| 🔍 矛盾检查 | 页面之间的矛盾声明 | ✅ |
| 📅 过时声明 | 被新资料取代的信息 | ✅ |
| 🚫 孤儿页面 | 没有入站链接的页面 | ✅ |
| 🏷️ 缺失页面 | 被提及但未创建的概念 | ✅ |
| 🔗 缺失交叉引用 | 应该存在的链接 | ✅ |
| 🌐 数据空白 | 可通过网络搜索填补的空白 | ⚠️ |

---

#### 4. File Back（回填）

**理念**：

> 好的答案不应该消失在聊天历史中

**机制**：

```
优质查询答案 → 归档为新Wiki页面 → 成为知识库一部分 → 未来查询可引用
```

**结果**：

```
你的探索过程本身也在为知识库复利增长
```

### 1.6 为什么有效？

#### 维护知识库的痛点

**不痛苦的部分**：
- ✅ 阅读
- ✅ 思考
- ✅ 提问

**痛苦的部分（记账工作）**：
- ❌ 更新交叉引用
- ❌ 保持摘要最新
- ❌ 标注矛盾
- ❌ 在数十个页面间保持一致性
- ❌ 重新组织结构

---

#### 人类为什么放弃

```
维护负担增长速度 > 价值增长速度
    ↓
放弃维护
    ↓
知识库退化为文档墓地
```

---

#### LLM的优势

**LLM不会**：
- 😴 感到无聊
- 🤦 忘记更新交叉引用
- 😰 因为需要修改15个文件而畏惧

**结果**：

```
维护成本 ≈ 0
    ↓
Wiki持续保持健康状态
    ↓
价值复利增长
```

---

#### 角色重新定义

| 角色 | 传统方式 | LLM-Wiki方式 |
|------|---------|-------------|
| **人类** | 阅读+思考+整理+维护 | 阅读+思考+策展+提问 |
| **AI** | 问答助手 | 知识库维护者 |
| **系统** | 被动存储 | 主动进化 |

### 1.7 历史渊源：Memex

#### Vannevar Bush的构想（1945）

**Memex**：一个个人化、策展式的知识存储系统，文档之间有关联线索（associative trails）。

**Bush的愿景**：

> "人脑通过联想工作...现有的索引系统是人工的。当数据被录入后，它只能通过预定义的路径提取。"

---

#### Memex的理想

- 🔒 私人的
- 🎨 主动策展
- 🔗 **连接与内容同等重要**
- 🧠 模仿人脑的联想思维

---

#### Bush无法解决的问题

> **谁来做维护？**

手工维护知识网络的成本太高，普通人无法持续。

---

#### LLM-Wiki的突破

```
Memex (1945)
  + 超文本技术 (1960s)
  + 个人Wiki工具 (2000s: Obsidian等)
  + 大语言模型 (2020s)
  ────────────────────────
  = LLM-Wiki (2026)
```

**关键创新**：
> LLM解决了维护问题，使Memex愿景终于可行

## 第二部分：LLM-Wiki Skill

### 2.1 项目概述

#### 项目定位

**LLM-Wiki Skill** 是一个**轻量级的Agent技能/插件**，为AI编程助手（如Claude Code、Codex、OpenClaw）提供知识库管理能力。

#### 基本信息

| 项目 | 信息 |
|------|------|
| **类型** | Agent技能/CLI工具 |
| **目标用户** | 个人开发者、研究人员 |
| **部署方式** | 本地安装 |
| **界面** | 无UI，通过Obsidian查看 |
| **数据存储** | 本地文件系统（Markdown） |
| **开源地址** | https://github.com/sdyckjq-lab/llm-wiki-skill |

---

### 2.2 核心理念

#### 设计哲学

1. **简单至上**
   - Bash脚本实现
   - 零数据库依赖
   - 纯文本Markdown文件

2. **工具集成**
   - 与Obsidian无缝配合
   - 与Git版本控制集成
   - 与现有工作流兼容

3. **多平台支持**
   - Claude Code
   - OpenAI Codex
   - OpenClaw

4. **内容丰富**
   - 支持9种内容来源
   - 覆盖主流信息渠道

#### 工具链详解

**1. baoyu-url-to-markdown**

```bash
# 安装
npm install -g baoyu-url-to-markdown

# 使用
baoyu-url-to-markdown https://example.com/article \
  --output raw-sources/web/article.md
```

**特性**：
- 提取正文内容
- 下载内联图片
- 清理广告和无关元素

---

**2. wechat-article-to-markdown**

```bash
# 安装
npm install -g wechat-article-to-markdown

# 使用
wechat-article-to-markdown https://mp.weixin.qq.com/s/xxx \
  --output raw-sources/wechat/article.md
```

**特性**：
- 保留微信文章格式
- 下载图片
- 提取作者和公众号信息

---

**3. yt-dlp + Whisper**

```bash
# 下载音频
yt-dlp -x --audio-format mp3 <youtube-url> -o audio.mp3

# 转录（使用OpenAI Whisper API或本地模型）
whisper audio.mp3 --model medium --language zh \
  --output_format txt \
  --output_dir raw-sources/youtube/
```

## Role
You are a Wiki Maintainer. Your job is to help build and maintain

## Directory Structure
- `raw-sources/`: Immutable source materials (read-only)

### 1. Ingest
When the user adds a new source:
1. Read the source file
2. Discuss key points with the user
3. Create a summary page in `wiki/sources/`
4. Update relevant entity pages in `wiki/entities/`
5. Update `wiki/index.md`
6. Add entry to `wiki/log.md`

### 2. Query
When the user asks a question:
1. Search `wiki/index.md` for relevant pages
2. Read those pages
3. Synthesize an answer with citations
4. Optionally, save the answer as a new wiki page

### 3. Lint
Periodically check wiki health:
- Look for contradictions
- Find orphan pages
- Identify missing cross-references
- Suggest new questions to explore

## Conventions
- Use `[[page-name]]` for internal links
- Add YAML frontmatter to pages:
  ```yaml
  ---
  created: 2026-04-08
  updated: 2026-04-08
  tags: [concept, ai, wiki]
  sources: [source-001]
  ---

- Always cite sources

## Tools Available

- `wechat-article-to-markdown <url>` - Download WeChat article
- `yt-dlp <url>` - Download YouTube video/audio

### C. 常见问题

#### Q1：两种实现可以互相迁移吗？

**A**：可以，但需要手动迁移

**从Skill到开源实现**：
1. 导出Markdown文件
2. 通过Web界面批量上传
3. 让Claude重新处理（批量导入）

**从开源实现到Skill**：
1. 导出所有Wiki页面
2. 整理到本地目录结构
3. 在Obsidian中打开

---

#### Q2：可以同时使用两种实现吗？

**A**：可以，推荐混合使用

**工作流**：
- Skill：快速记录、本地开发
- 开源实现：团队共享、Web访问

**同步**：
- 手动导出/导入
- Git同步
- API同步（需开发）

---

#### Q3：哪个实现更适合长期使用？

**A**：取决于你的场景

**个人长期**：Skill更可持续
- 数据完全控制
- 零运维成本
- 随时迁移

**团队长期**：开源实现更合适
- 协作功能
- 企业级特性
- 持续更新

---

#### Q4：隐私安全如何保障？

**Skill**：
- 数据本地存储
- Git加密
- 可使用本地LLM

**开源实现**：
- 自托管：完全控制
- 云端：依赖服务商
- RLS保护用户数据

   - 需要频繁回顾和连接？

3. **长期目标**
   - 计划持续使用 > 6个月？
   - 需要知识积累和复利？
