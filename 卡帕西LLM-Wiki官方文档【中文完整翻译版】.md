## 核心理念

### RAG的局限

大多数人使用LLM处理文档的体验类似于**RAG（检索增强生成）**：

```
你上传一系列文件 
  ↓
LLM在查询时检索相关片段
  ↓
生成答案
```

这种方式可以工作，但**LLM在每个问题上都从头开始重新发现知识**。

#### 问题所在

- ❌ **没有积累**（No accumulation）
- ❌ 每次都要重新检索和拼凑片段
- ❌ 需要综合5份文档的微妙问题？每次都要重新找、重新拼接
- ❌ 什么都没有被构建起来

**目前的RAG系统**（NotebookLM、ChatGPT文件上传、大多数RAG系统）都是这样工作的。

---

### LLM Wiki的不同之处

#### 核心差异

**不是**在查询时从原始文档检索，而是：

> LLM **增量式地构建和维护**一个持久化的Wiki——一个结构化、互相链接的Markdown文件集合，位于你和原始资料之间。

---

### 工作流程

#### 添加新资料时

```
传统RAG：索引它以便后续检索
   ↓
LLM Wiki：读取 → 提取关键信息 → 整合到现有Wiki中
```

**具体操作**：
- 更新实体页面
- 修订主题摘要
- 标注新数据与旧声明的矛盾之处
- 强化或挑战不断演进的综合信息

#### 关键特性

> **知识被编译一次，然后保持更新**——而不是在每次查询时重新推导。

---

### 核心区别：持久性与复利

这是关键差异：

```
Wiki是一个持久化的、复利性的制品
```

**优势**：
- ✅ 交叉引用已经存在
- ✅ 矛盾已经被标记
- ✅ 综合信息已经反映了你读过的所有内容
- ✅ Wiki随着你添加的每个资料和每个问题变得更加丰富

---

### 角色分工

#### 你的角色

- 📥 负责采集资料（sourcing）
- 🔍 负责探索（exploration）
- ❓ 负责提出正确的问题

#### LLM的角色

负责所有苦力活：
- 📝 摘要总结（summarizing）
- 🔗 交叉引用（cross-referencing）
- 📂 归档整理（filing）
- 📊 记账维护（bookkeeping）

> 这些工作让知识库随着时间推移真正有用。


>

**用途**：深入研究某个主题（数周或数月）

**资料来源**：

**你可以做什么**：
- 在阅读时个人构建类似的东西
- LLM完成所有交叉引用和维护工作

---

#### 4. 商业/团队

**用途**：由LLM维护的内部Wiki

**资料来源**：

#### 5. 其他场景

## 架构设计

### 三层架构

```
┌─────────────────────────────────┐
│   第一层：Raw Sources           │
│   (原始资料 - 不可变)            │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│   第二层：The Wiki              │
│   (LLM生成的Wiki - LLM拥有)     │
└─────────────────────────────────┘
              ↓
┌─────────────────────────────────┐
│   第三层：The Schema            │
│   (规则文档 - 共同进化)          │
└─────────────────────────────────┘
```

---

### 第一层：原始资料（Raw Sources）

#### 定位

你策展的源文档集合。

#### 内容类型

- 📄 文章（Articles）
- 📃 论文（Papers）
- 🖼️ 图片（Images）
- 📊 数据文件（Data files）

#### 特性

```
✓ 不可变（Immutable）
✓ LLM只读，从不修改
✓ 真相之源（Source of truth）
```

---

### 第二层：Wiki

#### 定位

一个包含LLM生成的Markdown文件的目录。

#### 内容类型

- 📝 摘要（Summaries）
- 🏷️ 实体页面（Entity pages）
- 💡 概念页面（Concept pages）
- ⚖️ 比较页面（Comparisons）
- 📋 概述（Overview）
- 🔗 综合信息（Synthesis）

#### 特性

> **LLM完全拥有这一层**

**LLM的工作**：
- 创建页面
- 当新资料到达时更新它们
- 维护交叉引用
- 保持一切一致性

**人类的工作**：
- 阅读它

---

### 第三层：Schema（规则文档）

#### 定位

一个配置文档，告诉LLM如何组织和维护Wiki。

#### 文件名示例

- `CLAUDE.md`（用于Claude Code）
- `AGENTS.md`（用于Codex）

#### 内容

告诉LLM：
- 📐 Wiki如何结构化
- 📋 有什么约定
- 🔄 在以下情况下遵循什么工作流：
  - 导入资料
  - 回答问题
  - 维护Wiki

#### 关键作用

> 这是关键的配置文件——**它让LLM成为一个有纪律的Wiki维护者，而不是一个通用聊天机器人**。

#### 进化特性

```
你和LLM随时间共同进化这个文档
   ↓
在你的领域中找出什么有效
```

## 操作流程

### 四大操作

```
1. Ingest  (导入)
2. Query   (查询)
3. Lint    (自检)
4. (隐含) File Back (回填)
```

---

### 1. Ingest（导入）

#### 操作描述

> 你将一个新资料放入原始资料集合，并告诉LLM处理它。

#### 典型流程示例

```
第1步：LLM读取资料
    ↓
第2步：与你讨论关键要点
    ↓
第3步：在Wiki中编写摘要页面
    ↓
第4步：更新索引
    ↓
第5步：更新Wiki中相关的实体和概念页面
    ↓
第6步：在日志中添加条目
```

#### 影响范围

> **单个资料可能会触及10-15个Wiki页面。**

---

#### 工作方式偏好

##### 个人方式（推荐）

**特点**：一次导入一个资料，保持参与

**工作流**：
- 阅读摘要
- 检查更新
- 指导LLM应该强调什么

##### 批量方式

**特点**：一次批量导入多个资料，较少监督

---

#### 自定义工作流

> **由你来开发适合你风格的工作流，并将其记录在Schema中供未来会话使用。**

---

### 2. Query（查询）

#### 操作描述

> 你向Wiki提问。

#### 工作流程

```
第1步：LLM搜索相关页面
    ↓
第2步：LLM读取它们
    ↓
第3步：综合答案并附上引用
```

---

#### 答案形式（根据问题而定）

| 形式 | 格式 |
|------|------|
| 📄 文档 | Markdown页面 |
| 📊 表格 | 比较表格 |
| 🎬 幻灯片 | Slide deck（Marp） |
| 📈 图表 | Chart（matplotlib） |
| 🎨 画布 | Canvas |

---

#### 重要洞察：回填机制

> **好答案可以被存回Wiki作为新页面。**

**可以回填的内容**：
- 你询问的比较
- 分析
- 你发现的连接

**为什么重要**：

> 这些都是有价值的，**不应消失在聊天历史中**。

**效果**：

```
你的探索像导入的资料一样，在知识库中复利增长
```

---

### 3. Lint（自检）

#### 操作描述

> 定期要求LLM对Wiki进行健康检查。

#### 检查项目

| 检查内容 | 说明 |
|---------|------|
| 🔍 **矛盾检查** | 页面之间的矛盾 |
| 📅 **过时声明** | 被新资料取代的陈旧声明 |
| 🚫 **孤儿页面** | 没有入站链接的孤儿页面 |
| 🏷️ **缺失页面** | 被提及但缺少自己页面的重要概念 |
| 🔗 **缺失交叉引用** | 应该存在但不存在的链接 |
| 🌐 **数据空白** | 可以通过网络搜索填补的数据空白 |

---

#### LLM的建议能力

LLM擅长建议：
- ❓ 需要调查的新问题
- 📚 需要寻找的新资料

**作用**：

> 随着Wiki的增长，保持其健康。

## 索引与日志

### 两个特殊文件

帮助LLM（和你）在Wiki增长时导航。它们服务于不同的目的。

---

### index.md（索引文件）

#### 定位

**面向内容**（Content-oriented）

#### 内容

Wiki中所有内容的目录：

**每个页面列出**：
- 🔗 链接
- 📝 一行摘要
- 🏷️ 可选的元数据（日期、资料数）

**组织方式**：
- 按类别组织（实体、概念、资料等）

---

#### 工作流程

```
第1步：LLM在每次导入时更新它
    ↓
第2步：回答查询时，LLM首先读取索引
    ↓
第3步：找到相关页面
    ↓
第4步：深入阅读它们
```

---

#### 扩展性

> 在中等规模（~100个资料，~数百个页面）下，这**效果出奇地好**。

**优势**：

```
避免了基于嵌入的RAG基础设施的需求
```

---

### log.md（日志文件）

#### 定位

**按时间顺序**（Chronological）

#### 内容

**仅追加的记录**（append-only record）：
- 发生了什么
- 何时发生

**记录事件**：
- 导入（Ingests）
- 查询（Queries）
- 自检通过（Lint passes）

---

#### 实用技巧

**一致的前缀**：

如果每个条目以一致的前缀开始（例如`## [2026-04-02] ingest | Article Title`），日志就可以用简单的Unix工具解析：

```bash
grep "^## \[" log.md | tail -5
```

**结果**：给你最后5个条目

---

#### 价值

- 📅 给你Wiki演化的时间线
- 🔍 帮助LLM理解最近完成了什么

### 2. 本地下载图片

#### 为什么重要

- 让LLM直接查看和引用图片
- 而不是依赖可能失效的URL

---

#### 配置步骤（Obsidian）

##### 步骤1：设置附件文件夹

```
Obsidian设置 → 文件和链接 → 附件文件夹路径
```

设置为固定目录（例如`raw/assets/`）

##### 步骤2：绑定快捷键

```
设置 → 快捷键 → 搜索"Download"
```

找到"Download attachments for current file"（下载当前文件的附件）

绑定到快捷键（例如Ctrl+Shift+D）

##### 步骤3：使用

```
剪辑文章后 → 按快捷键 → 所有图片下载到本地
```

---

#### 注意事项

**LLM的限制**：

> LLM无法在一次通过中原生读取带有内联图片的Markdown。

**解决方法**：

```
第1步：让LLM先读取文本
    ↓
第2步：然后单独查看部分或全部引用的图片
    ↓
第3步：获得额外的上下文
```

**评价**：

> 有点笨拙，但效果足够好。

### 关键讨论点

#### 1. 扩展性问题

**问题**：index.md在数千页面时会溢出上下文窗口

**解决方案**：
- 使用本地语义搜索（如OMEGA）
- 向量嵌入 + 全文搜索

---

#### 2. 知识形成机制

**讨论**：
- 什么值得成为记忆 vs. 什么是噪音？
- 如何知道存储的条目确实帮助了未来的会话？
- 需要反馈循环

---

#### 3. 数据溯源

**挑战**：
- 在组织间共享信息时
- 可能几个月后才发现坏情报
- 需要追踪"污染"源

**解决方案**：
- Git分支追踪
- 内容哈希
- 元数据记录

---

#### 4. 与Zettelkasten的关系

**相似点**：
- 都强调原子化笔记
- 都重视连接
- 都是网状结构

**区别**：
- Zettelkasten：人工维护
- LLM Wiki：AI维护
